PMO-Swarm

Multi-agent orchestrator for autonomous project management office operations. Scans Jira, audits RACI, enforces governance, generates daily operating briefs.

Built on ISRDS agentic platform. Production-ready. Runs on Google Cloud Run with Vertex AI Agent Engine.

What PMO-Swarm Does

PMO-Swarm runs on a schedule (every 60 minutes by default). Each cycle:

1. Scan

Execution Tracking Agent queries Jira for stalled tickets (>24h no activity)

2. Audit

Ownership/RACI Agent checks for missing owners or role gaps

3. Verify

Hygiene Agent scans for field violations, stale status

4. Draft

Follow-Up Agent drafts escalation messages for review

5. Report

Orchestrator synthesizes findings into Operating Brief

Output

Each cycle produces a daily Operating Brief:

📊 PMO OPERATING BRIEF — 2026-06-24 ✅ Execution Health • 3 stalled tickets (>48h) • ISRDS-1151: critical path, 169h no activity • ASHS-42: blocked on external vendor ⚠️ Governance & Escalations • Review gate: Follow-up message for ISRDS-1151 (pending approval) • RACI gap: 2 unassigned features in ROADMAP project 📈 Feature Progress • Mobile auth: 75% built (blocked on backend API) • Payment integration: awaiting schema finalization 📋 Team Notes • 5 tickets with missing Description field • 8 tickets stale in "In Progress" status >1 week

8-Layer Technical Model

PMO-Swarm's architecture is built in 8 distinct layers, each with a clear responsibility:

Layer Name Purpose
1 Config Registry Runtime config: model, Jira projects, scan interval
2 Agent Runtime Google ADK orchestrator, Runner, session management
3 Tools 22 tools: Jira, Firestore, AlloyDB, governance gates
4 Memory/Sessions ADK VertexAiSessionService for agent state
5 Governance Approval gates + Trust Ledger (audit trail)
6 Human Override Review/Approve/Escalate gates before Jira writes
7 Workflow Scheduled daemon loop, Cloud Run job trigger
8 Observability Cloud Logging, Monitoring, traces, metrics

Data Persistence: 4 Destinations

Each cycle writes to four storage layers for redundancy, analytics, and audit trails:

📄 Local Files

  • trust-ledger.jsonl
  • briefs/brief_*.txt
  • Append-only backup

☁️ Cloud Logging

  • Real-time searchable
  • Governance gates
  • Trace IDs for correlation

📊 BigQuery

  • operating_briefs table
  • cycle_metrics table
  • System of record

🗄️ AlloyDB

  • daily_briefings backup
  • Fast operational reads
  • 30-day retention

Quick Deploy Steps

Step 1: Build Docker Image

Package the agent code + dependencies

cd agents/pmo-swarm docker build -t pmo-agent:latest .

Step 2: Push to Artifact Registry

Store image in GCP for Cloud Run

docker tag pmo-agent:latest \ us-central1-docker.pkg.dev/isr-division-systems-488723/docker-repo/pmo-agent:latest docker push us-central1-docker.pkg.dev/isr-division-systems-488723/docker-repo/pmo-agent:latest

Step 3: Create Cloud Run Job

Deploy the job definition (one-time setup)

gcloud run jobs create pmo-swarm \ --image=us-central1-docker.pkg.dev/isr-division-systems-488723/docker-repo/pmo-agent:latest \ --region=us-central1 \ --project=isr-division-systems-488723 \ --timeout=3600

Step 4: Execute & Monitor

Run the job and watch logs

gcloud run jobs execute pmo-swarm --region=us-central1 --project=isr-division-systems-488723 # Stream logs gcloud logging read "resource.type=cloud_run_job AND resource.labels.job_name=pmo-swarm" \ --project=isr-division-systems-488723 --stream

Step 5: Set Up Scheduler (Optional)

Automate job execution on a cron schedule

gcloud scheduler jobs create app-engine pmo-scheduler \ --schedule="*/60 * * * *" \ --http-method=POST \ --uri="https://us-central1-run.googleapis.com/apis/run.googleapis.com/v1/projects/isr-division-systems-488723/locations/us-central1/jobs/pmo-swarm:run" \ --oidc-service-account-email=pmo-agent-svc@isr-division-systems-488723.iam.gserviceaccount.com \ --oidc-token-audience="https://us-central1-run.googleapis.com"

Health Check: What to Expect

  • Cloud Run job starts and shows "startup complete" log
  • 5 agents listed: execution_tracking, follow_up, ownership_raci, feature_completeness, hygiene
  • Orchestrator processes prompt and returns Operating Brief
  • Brief saved to briefs/brief_*.txt and Cloud Logging
  • BigQuery receives cycle_metrics and operating_briefs
  • AlloyDB daily_briefings table updated
  • Container exits with exit code 0

Production Hardening: Next Steps

PMO-Swarm is ready for production. To take it further:

1. Human Gates

Wire approval gates to Slack or PagerDuty. Implement gate_handler that waits for human approval before posting to Jira.

2. Alerts

Set up Cloud Monitoring alerts on governance escalations, stall counts, and feature gaps. Page on-call lead when critical.

3. Scale Agents

Add more skill agents: budget tracking, risk assessment, capacity planning, dependency analysis.

4. Retention Policy

Archive briefs to Cloud Storage after 30 days. Implement auto-cleanup of old AlloyDB and BigQuery records.

5. CI/CD Pipeline

Wire GitHub Actions → Artifact Registry → Cloud Run. Push-button deploy with approval gates.

6. Dashboard

Build a custom dashboard in Looker or Cloud Console. Surface briefs, stall trends, governance gate backlog.
